## 배운 점

### SQL — 자기 참조 관계 (중급 패턴 진입)
- 같은 테이블 두 번 JOIN — 다른 별칭으로 구분
- 자기 참조 관계 — 부모-자식, 매니저-직원 같은 트리 구조
- 3단계 JOIN 구조 — 관계가 여러 단계일 때
- 별칭 일관성 — SELECT의 컬럼 별칭 통일로 의도 명확히

### SQL — 조건 조합
- YEAR() + 범위 비교 (AGE >= 20 AND AGE <= 29)
- COUNT(*) — 조건에 맞는 행 개수

### Python — 문자열 곱셈
- `*` 연산자의 자료형별 의미 — 숫자 곱셈 vs 문자열 반복
- `"abc" * 3` = `"abcabcabc"`
- 이전 학습한 `+` 연산자(자료형별 의미)의 응용 패턴

### Python — 문자열 메소드 풍부함
- `swapcase()` — 대소문자 뒤집기
- `upper()`, `lower()` — 모두 대/소문자
- `capitalize()`, `title()` — 첫 글자 대문자
- `isupper()`, `islower()` — 대/소문자 판별
- 직접 구현 안 해도 — 메소드 활용으로 짧게

### Pythonic 사고법
"이걸 하는 메소드가 이미 있을까?" 먼저 생각.
- C 스타일: 메모리 주소·ASCII 코드 직접 다루기
- Python 스타일: 메소드로 추상화
- "어떻게"보다 "무엇을" 표현하는 데 집중

## 막힌 부분
- 자식 아이템 정보 가져오는 방법 — ITEM_INFO 두 번 JOIN 필요한 거 늦게 인식
- C 스타일 for문 `for(i in range(5)):` 작성 — Python은 괄호 없음
- "메모리 주소로 접근?" 사고 — Python은 메소드로 해결

## 자기 참조 관계 깊이 이해
이번 RARE 아이템 문제 — 본인이 처음 만난 self-referencing 패턴.
- A = 부모 정보 (ITEM_INFO)
- B = A의 자식 찾기 (ITEM_TREE)
- C = B의 자식 정보 (ITEM_INFO 다시)

같은 테이블을 다른 역할로 두 번 JOIN — 다른 별칭으로 구분.
